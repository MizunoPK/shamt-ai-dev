# Skill: shamt-code-review

## Overview

Produces a per-branch output folder under `.shamt/code_reviews/<sanitized-branch>/` with:
- `overview.md` — ELI5 + What/Why/How (formal reviews only; skipped for S7/S9)
- `overview_validation_log.md` — validation log for the overview
- `review_vN.md` — severity-tagged, copy-paste-ready review comments
- `review_validation_log.md` — validation log for the review

**Hard rule:** Never check out the branch. All git access is read-only.

---

## When This Skill Triggers

Use this skill when the user says any of: "review branch", "do a code review of",
"review the changes on", "run a code review", "re-review".

**Use formal workflow (with overview.md)** for:
- External PRs from teammates, other teams, or contributors
- Any branch you did not implement yourself

**Use S7/S9 variant (skip overview.md)** when:
- Running S7.P3 (Feature PR Review)
- Running S9.P4 (Epic PR Review)

**Do NOT start** if the branch cannot be fetched — halt immediately and tell the user.

## Protocol

### Step 1 — Access the Branch (Read-Only)

```bash
# Fetch if remote-only
git fetch origin <branch>

# Full diff vs merge base
git diff $(git merge-base main <branch>) <branch>

# List changed files
git diff --name-only $(git merge-base main <branch>) <branch>

# Read a specific file at branch tip
git show <branch>:<path/to/file>

# Commit log
git log main..<branch> --oneline
```

**If `git fetch` fails:** Halt. Report: "Branch `<branch>` could not be found locally or on
origin. Please verify the branch name and try again." Do not create any output files.

**Large diffs:** If the full diff is too large, read changed files individually via
`git show <branch>:<file>`.

**Model delegation:** Spawn Haiku sub-agent for git operations (mechanical, 70-80% token savings).

### Step 2 — Create Output Directory

Sanitize the branch name:
- Replace `/` with `_`
- Strip any character that is not alphanumeric, `-`, or `_`

Examples: `feat/my-feature` → `feat_my-feature`, `origin/feature/TICKET-99` → `origin_feature_TICKET-99`

Create `.shamt/code_reviews/<sanitized-branch>/` if it does not exist.

### Step 3 — Write `overview.md` (FORMAL REVIEWS ONLY; skip for S7/S9)

Create `.shamt/code_reviews/<sanitized-branch>/overview.md` using the template below.

**Header block:** Date, base branch, merge base SHA, commit range, file count.

**ELI5 — What Changed?**
- 2-4 sentences in plain English
- No jargon, no file or function names
- Focus on observable behavior, not implementation details
- Delegate to Sonnet sub-agent (summarization, not deep reasoning; ~40-50% savings)

**What Does This Branch Do?**
Clear statement of purpose and outcomes. What problem does it solve?

**Why Was It Built?**
Intent from commit messages, PR description, code reading. State "inferred from commit messages
/ code structure" when intent is not explicit.

**How Does It Work?**
Technical walkthrough: changed files, key logic, how components interact. Organize by area when
multiple subsystems are touched.

**Template:**
```markdown
# Branch Overview: <branch-name>

**Date:** YYYY-MM-DD
**Base:** main (merge base: <short SHA>)
**Commits:** N commits (<first SHA>..<last SHA>)
**Files changed:** N files (+X -Y lines)

---

## ELI5 — What Changed?

<2-4 sentences plain English, no jargon>

---

## What Does This Branch Do?

<purpose and outcomes>

## Why Was It Built?

<intent and motivation>

## How Does It Work?

<technical walkthrough>

---

*Overview generated by Shamt code review workflow*
*Branch: <branch> | Base: main | Date: YYYY-MM-DD*
```

### Step 4 — Overview Validation Loop (FORMAL REVIEWS ONLY; skip for S7/S9)

Create `overview_validation_log.md` BEFORE Round 1. On re-review, overwrite the existing log.

**5 dimensions — checked every round:**

| # | Dimension | Check |
|---|-----------|-------|
| 1 | Correctness | Every claim verified against diff/git show — not inferred from commit messages alone |
| 2 | Completeness | All files and hunks accounted for; no significant change silently omitted |
| 3 | Consistency | ELI5, What, Why, How sections agree; no internal contradictions |
| 4 | Clarity | ELI5 genuinely understandable to non-technical reader; How section specific enough |
| 5 | Scope | Describes only what is in the diff; no speculation about future changes |

**Round structure:**
1. Re-read entire `overview.md` from line 1 to end (partial reads do not count)
2. Check all 5 dimensions; document PASS or ISSUE for each
3. Fix any issues immediately in the file
4. State `consecutive_clean = N` at end of round

**Clean round:** ZERO issues OR exactly 1 LOW-severity issue (fixed).
**Counter resets to 0:** 2+ LOW issues OR any MEDIUM/HIGH/CRITICAL.

**Exit criteria:** After `consecutive_clean = 1`, spawn 2 parallel Haiku sub-agents to confirm zero
issues. Both must confirm → exit overview loop. Either finds an issue → fix, reset
`consecutive_clean = 0`, continue primary rounds.

### Step 5 — Determine Review Version Number

Check the branch sub-folder:
- No `review_v*.md` files → create `review_v1.md`
- `review_v1.md` exists → create `review_v2.md`
- And so on — **never overwrite a previous version**

### Step 6 — Write `review_vN.md`

Create `.shamt/code_reviews/<sanitized-branch>/review_vN.md`.

Check all 12 categories. Only include categories with actual findings — omit categories with no
issues. Group findings by severity (BLOCKING first, then CONCERN, SUGGESTION, NITPICK).

**12 Review Categories:**

| # | Category | Focus |
|---|----------|-------|
| 1 | Correctness | Logic errors, edge cases, off-by-ones — does the code do what the PR claims? |
| 2 | Code Quality | Naming, readability, duplication, unnecessary complexity |
| 3 | Comments & Documentation | Inline comments, docstrings, README updates |
| 4 | Code Organization | File structure, separation of concerns, function length, coupling |
| 5 | Testing | New tests added, coverage of new paths, test quality |
| 6 | Security | Input validation, auth checks, injection risks, sensitive data exposure |
| 7 | Performance | N+1 queries, unnecessary allocations, hot-path concerns |
| 8 | Error Handling | Missing error checks, swallowed exceptions, unclear failure modes |
| 9 | Architecture & Design | Pattern consistency with existing codebase, abstraction level |
| 10 | Backwards Compatibility | Breaking changes, API surface changes, deprecation handling |
| 11 | Scope & Changes | Does the PR match its stated purpose? Unrelated changes? |
| 12 | Context & Intent | (External PRs only) Does the author's intent appear sound? |

**Comment block format:**
```markdown
### [SEVERITY] Category N — <Category Name>

**File:** `path/to/file.ext`, line N

<Description: what line, what it does wrong, what the consequence is.>

**Suggested fix:** <Concrete direction for resolution.>
```

**Severity definitions:**

| Severity | Meaning |
|----------|---------|
| BLOCKING | Must be fixed before merge. Correctness bug, security vulnerability, data-loss risk. |
| CONCERN | Should be addressed. Quality, performance, or maintainability problem. |
| SUGGESTION | Optional improvement. Code works but could be better. |
| NITPICK | Minor style or preference. Author decides. |

**Template header:**
```markdown
# Code Review: <branch-name> (vN)

**Reviewed:** YYYY-MM-DD
**Base:** main (merge base: <short SHA>)
**Commits:** N commits (<first SHA>..<last SHA>)
**Files changed:** N files (+X -Y lines)
**Overview:** See `overview.md` for a full description of what this branch does.

---

## Review Comments

### BLOCKING
...

### CONCERN
...

### SUGGESTION
...

### NITPICK
...

---

## Validation Summary

**Rounds completed:** N
**Exit criteria:** Primary clean round + sub-agent confirmation
**Sub-agent A:** 0 issues
**Sub-agent B:** 0 issues
**Issues found and resolved during validation:** N
```

### Step 7 — Review File Validation Loop

Create `review_validation_log.md` BEFORE Round 1. On re-review, overwrite the existing log.

**Formal reviews: 12 dimensions. S7/S9 reviews: 13 dimensions.**

**Master Dimensions (7 — all contexts):**

| # | Dimension | Check |
|---|-----------|-------|
| 1 | Empirical Verification | Every comment cites specific file + line verified with git show or diff — not from memory |
| 2 | Completeness | Every file in the diff examined; no hunks or files silently skipped |
| 3 | Internal Consistency | No comments contradict each other; similar issues rated at same severity |
| 4 | Traceability | Each comment traces back to a specific diff hunk |
| 5 | Clarity & Specificity | Every BLOCKING and CONCERN names exactly what is wrong + concrete fix |
| 6 | Codebase Alignment | Comments don't flag patterns already established in the existing codebase |
| 7 | Format Compliance | Follows output_format.md — severity tags, file references, section structure correct |

**Code Review-Specific Dimensions (5 standard):**

| # | Dimension | Check |
|---|-----------|-------|
| 8 | Coverage Balance | Comment distribution reflects actual risk — critical areas not under-reviewed |
| 9 | False Positive Audit | Every BLOCKING/CONCERN re-verified against current diff |
| 10 | Actionability | Every comment gives author enough context to make a concrete change |
| 11 | Severity Calibration | BLOCKING reserved for genuine merge blockers; distinctions consistently applied |
| 12 | Scope Coherence | Comments stay within what the PR is attempting; no scope creep |

**S7/S9-Specific Dimension (only when invoked from S7.P3 or S9.P4):**

| # | Dimension | Check |
|---|-----------|-------|
| 13 | Implementation Fidelity | Every proposal in the validated implementation plan has corresponding code; all spec requirements addressed; no scope creep; no missing features |

**Adversarial Self-Check (required before declaring any round clean):**
> "What would a thorough human reviewer catch that I have not flagged?"

Re-read the diff from the perspective of a skeptic. A round may NOT be scored clean if skipped.

**Adversarial Linter Check (required before declaring any round clean):**
> "What would ESLint/Ruff/CodeQL flag that I haven't checked?"

Verify: unused variables/imports, operator confusion, missing null checks, unreachable code,
string quote inconsistencies, type coercion, security patterns (eval, innerHTML, SQL concat).

**Round structure:**
1. Re-read entire `review_vN.md` from line 1 to end
2. Re-verify at least 3 specific technical claims against `git show` or diff (document the calls)
3. Check all dimensions (12 or 13); document PASS or ISSUE for each
4. Run Adversarial Self-Check and Adversarial Linter Check
5. Fix any issues immediately in the review file
6. State `consecutive_clean = N` at end of round

**Exit criteria:** After `consecutive_clean = 1`, spawn 2 parallel Haiku sub-agents confirming zero
issues. Both confirm → exit. Either finds an issue → fix, reset `consecutive_clean = 0`, continue.

## S7/S9 Variant — Key Differences

| Aspect | Formal | S7/S9 Variant |
|--------|--------|---------------|
| overview.md | Required (Steps 3-4) | SKIPPED |
| Dimensions | 12 | 13 (+ Implementation Fidelity) |
| Step sequence | 1→2→3→4→5→6→7 | 1→2→5→6→7 |
| Reviewer | Primary agent (Opus) | Fresh Opus sub-agent (zero implementation bias) |

**S7.P3 scope:** Feature-level code quality.
**S9.P4 scope:** Epic-level concerns only (cross-feature integration, architectural coherence) —
per-feature code quality was already handled in S7.P3.

**S7/S9 comment addressing protocol:**
- BLOCKING: Fix immediately (correctness, security, data-loss)
- CONCERN: Fix immediately (real quality/performance/maintainability problem)
- SUGGESTION: Walk through with user one-by-one; user decides fix/document/escalate
- NITPICK: Walk through with user one-by-one; user decides fix/document/escalate

---

## Exit Criteria

**Overview validation:** `consecutive_clean = 1` (primary) + 2 Haiku sub-agents confirm zero issues.

**Review validation:** Same — `consecutive_clean = 1` (primary) + 2 Haiku sub-agents confirm zero issues.

**Clean round definition:**
- ZERO issues found, OR
- Exactly 1 LOW-severity issue found AND fixed

**Not clean:** 2+ LOW issues OR any MEDIUM/HIGH/CRITICAL. Resets counter to 0.

**Sub-agent exception:** Sub-agents do NOT get the 1 LOW allowance. Any issue found by a
sub-agent resets the counter to 0.

## Quick Reference

**Model delegation pattern:**
```
Primary Agent (Opus):
├─ Spawn Haiku → Git operations (fetch, file list, commit messages)
├─ Spawn Sonnet → overview.md ELI5 section
├─ Primary → Issue classification, adversarial checks, review_vN.md
├─ Spawn 2x Haiku (parallel) → Overview validation sub-agent confirmation
└─ Spawn 2x Haiku (parallel) → Review validation sub-agent confirmation
```

**Output files per review:**
```
.shamt/code_reviews/<sanitized-branch>/
├── overview.md                (formal reviews only)
├── overview_validation_log.md (formal reviews only)
├── review_v1.md
├── review_validation_log.md
└── review_v2.md               (re-reviews only)
```

**Anti-shortcuts:**
- Never check out the branch
- Never overwrite review_v1.md — always create review_v2.md
- Never skip the adversarial self-check before declaring a round clean
- Never skip Dimension 13 for S7/S9 reviews
- Do not create overview.md for S7/S9 reviews
