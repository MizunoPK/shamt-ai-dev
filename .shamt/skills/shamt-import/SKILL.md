## Overview

This skill runs **after** the user executes `import.sh` (or `import.ps1`). The script handles file sync mechanically; this skill covers the validation work required afterward.

The import script has already:
- Copied updated/new files from master's `guides/`, `scripts/`, `skills/`, `agents/`, `commands/`, and `hooks/` into your `.shamt/`
- Removed files deleted from master (only those that came from master)
- Preserved any child-only files (never in master) without deleting them
- Generated diff file(s): `.shamt/import_diff.md` (or `import_diff_1.md`, `import_diff_2.md`, etc.)
- If `ai_service.conf` is `claude_code`, `codex`, or `claude_codex`: ran the matching regen script to refresh `.claude/`/`.codex/` artifacts and the project-tailored `.shamt/CHEATSHEET.md` (transparent — no agent action required)

Your job: assess the impact and ensure project-specific configuration stays consistent.

## Protocol

### Mandatory First Step: Re-Read the Validation Loop Protocol

**Before doing anything else**, re-read:

`.shamt/guides/reference/validation_loop_master_protocol.md`

You will run a validation loop (primary clean round + sub-agent confirmation required) after completing the assessment below.

---

### Step 1: Read the Import Diff

Read all diff files generated by the import script:
- If `.shamt/import_diff.md` exists, read it
- If `.shamt/import_diff_1.md`, `import_diff_2.md`, etc. exist, read all in order

The diffs show exactly what changed (child content → master content). Read them carefully before making any decisions.

> Model delegation: Spawn Haiku to read diff files and count changed files.

---

### Step 2: Check Supplements for Each Changed File

For each file shown in the diff:

1. Does this project have a supplement in `.shamt/project-specific-configs/guides/[mirrored-path]/`?
2. If yes — does the supplement still make sense given the new guide content?
3. Is the supplement still accurate, or did the master update change the context it was written for?

Update any supplements that are now inaccurate or inconsistent with the new guide content.

> Model delegation: Spawn Sonnet to read supplement files and check consistency.

---

### Step 3: Check Pointers

For each changed file:

1. Are any pointer notes in the updated guide file still accurately placed?
2. If a section was moved, renamed, or removed — the pointer may reference the wrong location
3. Update or remove stale pointers

Also check whether the new guide content introduced new extension points that would benefit from a pointer + supplement for this project. If so, create the supplement and add the pointer.

---

### Step 4: Check for New Files and Deletions

The import summary lists any newly added or deleted files.

**For new files from master:**
- Read them to understand what they add
- Assess whether this project needs a `project-specific-configs/` supplement

**For deleted files:**
- Confirm the deletion makes sense (guide was retired or merged)
- Check if any `project-specific-configs/` supplements referenced the deleted guide — if so, update or remove them

---

### Step 4.2: Apply New Sections from RULES_FILE.template.md

If `scripts/initialization/RULES_FILE.template.md` appears in the import diff:

1. Read the updated template
2. Compare it against your project's rules file (e.g., `CLAUDE.md`)
3. Identify sections present in the new template but absent from your rules file
4. Apply those new sections to your rules file — preserving all project-specific content

This is how new agent capabilities (new trigger phrases, new workflow steps) reach existing child projects after an import. The template is not a guide; supplement/pointer logic does not apply to it.

If no sections are missing, note "Rules file compared — no template updates needed" and proceed.

---

### Step 4.5: Review Preserved Child-Only Files

If the import script reported preserved files ("child-only files not deleted"), review each one:

- **Planned for export to master:** Confirm it is in `.shamt/CHANGES.md`. Keep it in `guides/`. It will export via the normal export workflow.
- **Correctly project-specific:** Move it to `.shamt/project-specific-configs/guides/[mirrored-path]/`. Add a pointer to the relevant shared guide if appropriate.
- **Obsolete:** Delete it manually.

---

### Step 5: Run a Validation Loop

Run a full validation loop. Track `consecutive_clean` explicitly (start at 0).

**Each round — check for:**
- All supplements still accurate and consistent with updated guides
- All pointers correctly placed and not stale
- No project-specific content accidentally introduced into shared guides
- No broken cross-references in updated guides
- `SHAMT-{N}` placeholder intact throughout `.shamt/guides/` — NEVER replace it with your project's epic tag

**On the `SHAMT-{N}` placeholder:** It appears throughout shared guides as a generic placeholder for "epic number N." It applies equally to every child project. If you find any substitutions (e.g., `KAI-{N}`, `BAR-{N}`), revert them immediately before the validation loop can pass.

**Clean round**: ZERO issues OR exactly ONE LOW-severity issue (fixed). Two or more LOW issues OR any MEDIUM/HIGH/CRITICAL resets `consecutive_clean` to 0.

**When `consecutive_clean = 1` → spawn 2 Haiku sub-agents in parallel:**

- Both confirm zero issues → proceed to Step 6
- Either sub-agent finds issues → fix all, reset `consecutive_clean = 0`, continue loop

---

### Step 6: Delete the Diff Files

When the validation loop achieves primary clean round + sub-agent confirmation, delete all import diff files:

```bash
rm -f .shamt/import_diff.md .shamt/import_diff_*.md
```

Do not leave diff files in place after import is complete — they are temporary and contain dated content.

---

### Step 7: Commit

Stage and commit the results:

```bash
git add .shamt/guides/ .shamt/scripts/ .shamt/skills/ .shamt/agents/ .shamt/commands/ .shamt/hooks/ .shamt/project-specific-configs/
git commit -m "sync: Import Shamt updates from master (YYYY-MM-DD)

[Brief description of what changed]"
```

Include only files that were part of the import or supplement updates. Do not bundle other work into this commit.

---

## Exit Criteria

Import workflow is complete when ALL of the following are true:

- [ ] All import diff files read and understood
- [ ] All changed-file supplements reviewed and updated where needed
- [ ] All pointers checked and updated where stale
- [ ] New files assessed for supplement needs
- [ ] Deleted files' supplement references cleaned up
- [ ] RULES_FILE template compared; new sections applied to rules file if present
- [ ] Preserved child-only files reviewed and routed correctly
- [ ] Validation loop passed: primary clean round + both sub-agents confirmed zero issues
- [ ] All import diff files deleted
- [ ] Changes committed with "sync: Import Shamt updates" message

## Quick Reference

### Common situations

**No diff files, only new/deleted/preserved files:**
Skim new files, check deletions, review preserved files per Step 4.5. Validation loop still required, typically shorter.

**Large import with many changed files:**
Work through one file at a time. Diffs are split into numbered files so each is readable in one pass. Prioritize high-traffic guides (S5, S7, S10).

**Changed guide conflicts with a project-specific supplement:**
The supplement takes precedence for this project. But consider whether the master update is better — if so, update the supplement to align.

**Import script fails with "Master directory not found":**
`.shamt/shamt_master_path.conf` is stale. Update with current path:
```bash
echo "/path/to/shamt-ai-dev" > .shamt/shamt_master_path.conf
```

**Your `.shamt/guides/` is gitignored:**
Remove it from `.gitignore`. Only `.shamt/*.conf` and `.shamt/import_diff*.md` should be gitignored.

### Model selection

- Reading diff files, counting changed files: Haiku
- Reading supplements, checking pointer consistency: Sonnet
- Impact assessment, validation loop decisions: Opus (primary)
- Sub-agent confirmations: Haiku (always)
