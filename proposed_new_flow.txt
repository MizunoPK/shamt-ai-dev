- A user will clone the shamt-ai-dev repo onto their system
- The user will run the initialization script and provide basic information such as the path to the project that should be init with shamt, AI agent, epic tag, etc.
- A .shamt folder will be created by the script in the target project and the files from the master folder will be copied over
- The script will then provide a prompt that the user should provide their AI agent in the project. Something along the lines of "Read `.shamt/project-specific-configs/init_config.md` and complete the Shamt initialization."
- The agent should then finalize the initilation of the shamt configs by analyzing the repo and asking the user any remaining questions to better understand the project, and create the final rules file, architecture file, and coding standards file unique to the project. This should be done via a validation loop according to the validation loop protocol to ensure the highest quality files possible.
- The 'project-specific-configs' folder should be made in the .shamt folder. This should be where any project-specific information lives. Any modifications to guide flow, special project-specific information, or project-specific rules files should be kept there.
- Throughout initialization, S10 lessons learned updates, and audit updates... if there is any information to be saved or updated that is specific to the child-Shamt's project then only files in the project-specific-configs folder should be created or updated. If a specific file from the original guides needs to be supplemented with extra project-specific information that is included in the project-specific-configs folder, then the relevant files outside of that folder should be updated to direct the agent to the project-specific-configs folder to see if there is any supplemental information because that section has been supplemented in the past.
- Any changes made to the non-project specific files during the S10 updates, audit, or supplemental info notes, should be noted in a changelog file. We should maintain a changelog file in the .shamt folder that notes updates made to the main files.
- There should exist an export script that can be used to copy the updated non-project specific files that were created or modified back to the shamt-ai-dev repo that should exist locally on the user's computer. This would automate applying the changes to the master repo and omit the need for passing changelogs between versions of Shamt. It would omit moving over the changes made to any files in the project-specific-configs folder.
- The user would then make a branch of the shamt-ai-dev project and create a PR into the main branch to ensure the master version of Shamt recieves the latest updates
- When another user is looking to update their version of child-Shamt with the master version, then they would pull the latest changes in their local copy of the shamt-ai-dev repo, then run an 'import' script that would copy over any new files into the target project and update any modified files.
- The import script should end with providing a prompt to tell the user's AI agent which will tell the agent to analyze the latest changes from the master version of Shamt and perform a validation loop based on the validation loop protocol to ensure the project-specific files are updated to accomidate the changes if required
